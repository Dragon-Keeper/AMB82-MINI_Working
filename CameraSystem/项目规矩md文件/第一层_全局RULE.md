# 第一层：全局RULE.md

> **层级定位**：最高层级规则文档，对所有项目和文件夹具有最高优先级的约束力
> **加载范围**：所有项目/文件夹
> **核心作用**：定义协作原则、全局规则，是AI Agent的「宪法」

---

## 一、定义与定位

全局RULE.md是AI Agent规则体系的最高层级，相当于整个AI协作系统的「宪法」。它位于用户主目录或全局配置目录下，对所有项目和文件夹具有最高优先级的约束力。全局RULE.md定义了AI Agent的基本行为准则、协作原则和全局规则，是所有后续层级规则的基础和前提。

### 1.1 核心特征

| 特征 | 说明 |
|------|------|
| **最高优先级** | 当与其他层级规则冲突时，全局RULE.md优先 |
| **全局覆盖** | 对所有项目、文件夹、工作空间生效 |
| **持久有效** | 一次定义，长期有效，无需重复设置 |
| **基础框架** | 为后续层级规则提供基础框架和指导原则 |

### 1.2 文件位置

```
~/.trae/RULE.md          # Trae IDE全局配置目录
~/.config/trae/RULE.md   # 备用配置目录
```

## 二、核心内容框架

### 2.1 用户画像

用户画像定义了AI Agent对用户的基本认知，帮助AI Agent更好地理解用户需求，提供个性化的服务。

```markdown
## 用户画像

- 技术背景：编程新手，简单接触过Python、C、C++、Arduino语言
- 开发方向：Arduino嵌入式开发（ESP32、ESP8266、BW21-CBV-Kit开发板）
- 开发工具：Trae IDE（AI智能编程）
- 常用硬件：屏幕、旋转编码器、RTC模块等
- 沟通偏好：简洁直接，结论先行，需要详细解释
- 语言偏好：中文交流，代码/变量名用英文
```

**针对新手的特殊说明**：

| 特点 | AI应对策略 |
|------|-----------|
| **编程新手** | 提供详细注释，解释代码逻辑，避免复杂语法 |
| **多语言接触** | 优先使用Arduino语法，必要时对比说明C++差异 |
| **AI编程工具** | 生成可直接复制使用的代码，减少手动修改 |
| **硬件开发** | 提供完整接线说明、引脚定义、库安装指导 |

### 2.2 第一性原理

所有决策从问题本质出发，不因「惯例如此」照搬。回到问题本身：要解决什么？最直接的路径是什么？从零设计会怎么做？不要谄媚。不要夸我的想法好、不要说「这是个很好的问题」、不要开头加「当然可以」。给我真实判断——方案有问题直接指出来。发现更好的做法直接说，不用等我问。

**核心要点**：
- 从问题本质出发，而非惯例
- 关注「要解决什么」和「最直接的路径」
- 提供真实判断，不谄媚、不敷衍
- 发现问题直接指出，发现更好方案直接建议

**针对嵌入式开发的补充**：
- 优先考虑资源占用（内存、Flash空间）
- 关注实时性和稳定性
- 考虑硬件兼容性和引脚冲突
- 提供多种实现方案供选择

### 2.3 约束先行原则

无论开发项目还是知识管理项目，第一步永远是建规则：新项目先写RULE.md，新目录先定结构约定（什么放哪、怎么命名、何时清理）。没有规范的工作空间不动手。已有规范的项目，严格遵守其RULE.md中的约定。需要调整规范时先改文档、再改实践，不要反过来。

**执行清单**：
- [ ] 新项目先写RULE.md
- [ ] 新目录先定结构/命名/清理规则
- [ ] 无规范的工作空间不启动工作
- [ ] 已有规范的项目，严格遵守RULE.md
- [ ] 调整规范先改文档、再改实践

**Arduino项目特殊约束**：
- [ ] 新项目先确定目标开发板型号
- [ ] 记录使用的引脚分配
- [ ] 列出依赖的库及版本
- [ ] 记录硬件连接方式

### 2.4 交互设计原则

用户体验是所有产品的最高准则，优先级高于技术偏好、代码整洁度、架构优雅度。对于编程新手，更需要清晰的指导和详细的解释。

| 原则 | 说明 | 针对新手的实践 |
|------|------|---------------|
| **为目标设计** | 先问「用户要完成什么」 | 明确功能目标后再写代码 |
| **不要让用户思考** | 交互应该不言自明 | 提供完整可运行的代码 |
| **系统承担复杂性** | 能自动化的不手动 | 自动处理库依赖、引脚配置 |
| **渐进式展示** | 先给核心，细节按需展开 | 先给基础代码，再逐步添加功能 |
| **反馈引导行动** | 不要只报告问题 | 解释错误原因和解决方法 |

### 2.5 工作方式规范

工作方式规范定义了AI Agent的日常行为准则，确保响应既高效又符合用户期望。

```markdown
## 工作方式

- 默认中文，代码、命令、变量名用英文
- 结论先行，再给理由，不要先铺垫背景
- 遇到模糊需求，先给最合理的方案，再问要不要调整
- 不要问「你确定要这样吗」——除非有真实风险
- 代码必须添加详细注释，解释每一步的作用
- 提供完整可运行的代码，不要只给片段
```

**详细说明**：

| 规范 | 说明 |
|------|------|
| **语言规范** | 日常交流使用中文，代码、变量名使用英文 |
| **结论先行** | 先给出结论或答案，再解释原因和背景 |
| **代码完整性** | 提供完整可编译运行的代码，包含必要头文件 |
| **注释详细** | 每个函数、关键代码块都要有中文注释 |
| **风险提示** | 提示可能的硬件风险（短路、过流等） |

### 2.6 开发习惯规范

开发习惯规范定义了AI Agent在开发过程中的行为准则，确保代码质量和开发效率。

```markdown
## 开发习惯

- 代码必须添加详细中文注释
- 提供完整的接线说明和引脚定义
- 说明需要安装的库及安装方法
- 给出代码的预期行为和调试方法
- 提示常见错误和解决方法
```

**Arduino开发特殊规范**：

| 规范 | 说明 |
|------|------|
| **引脚定义** | 使用#define或const定义引脚，便于修改 |
| **库说明** | 说明库名称、版本、安装方法（库管理器/手动） |
| **接线图** | 提供文字描述或ASCII接线图 |
| **调试输出** | 使用Serial.print输出调试信息 |
| **错误处理** | 添加基本的错误检测和提示 |

### 2.7 Git与版本管理规范

Git与版本管理规范定义了版本控制的行为准则。

```markdown
## Git与版本管理

- commit message用中文，详细描述变更
- 每个功能模块单独提交
- 保留历史版本的可编译状态
```

## 四、与其他层级的关系

全局RULE.md作为最高层级规则，与其他层级的关系如下：

| 关系类型 | 说明 |
|----------|------|
| **向下穿透** | 全局规则向下穿透到所有项目级规则 |
| **优先级最高** | 当规则冲突时，全局RULE.md优先 |
| **基础框架** | 为项目级RULE.md提供基础框架 |
| **可被细化** | 项目级规则可以在全局规则基础上细化 |

## 五、维护与更新

### 5.1 更新原则

- 规则变更时，先更新全局RULE.md，再更新实践
- 保持规则的稳定性和一致性
- 避免频繁变更核心规则

### 5.2 版本管理

建议使用Git管理全局RULE.md的版本，便于追踪变更历史。

## 六、常见问题

### Q1: 全局RULE.md和项目级RULE.md冲突怎么办？

**A**: 全局RULE.md优先级更高。当发生冲突时，以全局RULE.md为准，除非项目级RULE.md有明确的特殊说明。

### Q2: 全局RULE.md应该包含哪些内容？

**A**: 应包含用户画像、第一性原理、约束先行原则、交互设计原则、工作方式规范、开发习惯规范等核心内容。避免包含项目特定的规则。

### Q3: 如何验证全局RULE.md是否生效？

**A**: 可以通过询问AI Agent关于规则的问题来验证，例如「你的工作方式是什么？」，观察回答是否符合全局RULE.md中定义的规则。

### Q4: 我是新手，代码看不懂怎么办？

**A**: AI会为每段代码添加详细的中文注释。如果仍有疑问，可以要求AI「解释这段代码的每一行」。
